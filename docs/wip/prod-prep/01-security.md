# Arkfile Live Security and Privacy Gaps

This document is the in-codebase security and privacy work that genuinely remains open as of 2026-06-08. It was derived from a code-verified review: every item below was checked against the live source tree on this date, and each item states whether it is a genuinely exploitable gap or defense-in-depth hygiene, so prioritization does not have to be re-litigated. This is intentionally separate from the deployment/operational hardening checklist (see `02-prod-hardening.md`) and the post-launch feature roadmap (see `03-roadmap.md`). The older mixed-scope inventory at `docs/wip/review/remaining-issues.md` is left in place as the historical baseline; this document supersedes its in-codebase-security portion.

## The one genuinely exploitable gap: the app binds all interfaces

This is the single most important item and the one fix worth doing before anything else. The Arkfile process itself listens on all interfaces. In `main.go` the server is started with `e.StartTLS(":"+tlsPort, certFile, keyFile)` (around line 345) and `e.Start(":" + port)` (around line 357), and neither call constrains the bind address to a host. The deployment design intends Caddy to be the only public face, terminating TLS and reverse-proxying to Arkfile over loopback, but with the current bind the Arkfile process is directly reachable on every interface of the host. If the firewall is ever absent, disabled, misconfigured, or changed, anyone on the internet can talk straight to the application on ports 8080/8443, bypassing Caddy entirely and exposing the app's own HTTP/TLS stack. This is a real, internet-facing exposure rather than hygiene, which is why it leads this list.

There is an important refinement that makes this smaller than it first appears, and the fix should be written with this in mind. The configuration field already exists: `config.Server.Host` is defined in `config/config.go`, defaults to `"localhost"` in `loadDefaultConfig`, and is populated from the `HOST` environment variable in `loadEnvConfig` (the `HOST` env var is already read elsewhere in `main.go` for the legacy `TEST_PORT` logic, so it is a live, wired value). The gap is therefore two narrow points, not a new subsystem. First, `main.go` ignores `cfg.Server.Host` at bind time and uses a bare `":port"`; it should bind to `cfg.Server.Host + ":" + port` (and the same for the TLS listener), so a configured host actually constrains the listener. Second, the production deploy path does not write a `HOST` value, so even after `main.go` honors it, production would still default to `localhost` only by accident of the default rather than by explicit configuration; `prod-deploy.sh` should write `HOST=127.0.0.1` into `secrets.env`. This is consistent with the already-completed work that bound rqlite and SeaweedFS to loopback; the application is currently the one service still listening on `0.0.0.0`, and closing that is the natural completion of that hardening. Care is needed for the local/LAN deploy case, where binding strictly to `127.0.0.1` would prevent LAN access; that path can legitimately set `HOST=0.0.0.0` (or a LAN address) while production sets loopback, so the behavior must remain configurable rather than hardcoded to loopback in the binary.

## Client-side padding so the server never sees the unpadded size

Today the server applies random padding to the last chunk of an upload in `handlers/uploads.go`, which means the server observes the true unpadded ciphertext size before it pads. This is a real privacy improvement to pursue, not mere hygiene, because Arkfile's stated promise is that the server (and any third-party S3 backend) learns nothing about the nature of a user's data, and exact ciphertext size is a metadata signal. The reason it is not the first priority is cost and blast radius: moving padding to the client is a Medium-effort, coordinated change that touches the upload pipeline and must be mirrored across the Go server, the TypeScript frontend, and the `arkfile-client` CLI so that all clients pad identically and the server validates accordingly. It is worth scheduling as its own deliberate session rather than squeezing in alongside the loopback fix, and it must respect the constrained-device streaming model (padding strategy cannot require buffering a whole file in memory).

## Consciously deferred: metadata-encryption key domain separation

The same Account-KEK that wraps per-file FEKs also directly encrypts filename and SHA-256 metadata, with no HKDF-derived metadata subkey to isolate the two roles. This was reviewed and consciously deferred, and that judgment still holds: it is defense-in-depth hygiene, not an exploitable hole. It does not defend against a cracked Account Key, because breaking the Argon2id derivation already yields the master secret that both unwraps FEKs and decrypts metadata, so a dedicated subkey buys no additional protection against that threat. Implementing it would also require a coordinated encrypted-envelope format change across the Go server, the TypeScript frontend, and the CLI for marginal benefit. The right time to do this is opportunistically, when a crypto-hardening session is already touching these envelope paths, not as a standalone effort.

## Lower-leverage residuals to revisit later

Two smaller items remain genuinely useful but lower-leverage. The anonymous EntityID rotates daily (`logging/entity_id.go`), so a per-EntityID rate-limit budget resets each day and an attacker can multiply their budget by waiting out the rotation window; this is a privacy-versus-abuse tradeoff (a longer-lived EntityID resists this abuse but is a slightly stronger long-term identifier) rather than a confidentiality leak, and it is worth a deliberate decision rather than a reflexive change. Usernames are also compared byte-wise with no Unicode normalization or case-folding policy (`utils/username_validator.go` and comparison sites), so visually identical usernames with different byte encodings are treated as distinct, which is an impersonation-adjacent confusion risk rather than a direct compromise. Both can be slotted into a later session once the loopback bind is closed and the deployment hardening in `02-prod-hardening.md` is underway.

## Recommended sequence

Do the loopback bind first, because it is the only item here that closes an internet-facing exposure and it is small now that `config.Server.Host` already exists. Schedule client-side padding next as its own coordinated cross-client change. Treat the metadata-subkey separation as opportunistic work tied to a future crypto session, and slot the EntityID-rotation tradeoff and username normalization into later cleanup. None of the deferred items should displace the loopback fix.
