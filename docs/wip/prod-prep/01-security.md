# Arkfile Live Security and Privacy Gaps

This document is the in-codebase security and privacy work that genuinely remains open as of 2026-06-08. It was derived from a code-verified review: every item below was checked against the live source tree on this date, and each item states whether it is a genuinely exploitable gap or defense-in-depth hygiene, so prioritization does not have to be re-litigated. This is intentionally separate from the deployment/operational hardening checklist (see `02-prod-hardening.md`) and the post-launch feature roadmap (see `03-roadmap.md`). The older mixed-scope inventory at `docs/wip/archive/review/remaining-issues.md` is left in place as the historical baseline; this document supersedes its in-codebase-security portion.

## The one genuinely exploitable gap: the app binds all interfaces

This is the single most important item and the one fix worth doing before anything else. The Arkfile process itself listens on all interfaces. In `main.go` the server is started with `e.StartTLS(":"+tlsPort, certFile, keyFile)` (around line 345) and `e.Start(":" + port)` (around line 357), and neither call constrains the bind address to a host. The deployment design intends Caddy to be the only public face, terminating TLS and reverse-proxying to Arkfile over loopback, but with the current bind the Arkfile process is directly reachable on every interface of the host. If the firewall is ever absent, disabled, misconfigured, or changed, anyone on the internet can talk straight to the application on ports 8080/8443, bypassing Caddy entirely and exposing the app's own HTTP/TLS stack. This is a real, internet-facing exposure rather than hygiene, which is why it leads this list.

There is an important refinement that makes this smaller than it first appears, and the fix should be written with this in mind. The configuration field already exists: `config.Server.Host` is defined in `config/config.go`, defaults to `"localhost"` in `loadDefaultConfig`, and is populated from the `HOST` environment variable in `loadEnvConfig` (the `HOST` env var is already read elsewhere in `main.go` for the legacy `TEST_PORT` logic, so it is a live, wired value). The gap is therefore two narrow points, not a new subsystem. First, `main.go` ignores `cfg.Server.Host` at bind time and uses a bare `":port"`; it should bind to `cfg.Server.Host + ":" + port` (and the same for the TLS listener), so a configured host actually constrains the listener. Second, the production deploy path does not write a `HOST` value, so even after `main.go` honors it, production would still default to `localhost` only by accident of the default rather than by explicit configuration; `prod-deploy.sh` should write `HOST=127.0.0.1` into `secrets.env`. This is consistent with the already-completed work that bound rqlite and SeaweedFS to loopback; the application is currently the one service still listening on `0.0.0.0`, and closing that is the natural completion of that hardening. Care is needed for the local/LAN deploy case, where binding strictly to `127.0.0.1` would prevent LAN access; that path can legitimately set `HOST=0.0.0.0` (or a LAN address) while production sets loopback, so the behavior must remain configurable rather than hardcoded to loopback in the binary.

## Usernames in Arkfile

The remaining-issues baseline expressed concern regarding Unicode normalization, case-folding, and collision vulnerabilities. A rigorous review of `utils/username_validator.go` and the authentication paths reveals custom properties and hidden constraints that dictate specific design responses for the Arkfile architecture.

First, visual homograph attacks via Unicode characters are not a threat in Arkfile because the validator physically blocks all non-ASCII inputs. Characters are restricted strictly to `[a-zA-Z0-9_\-.,]` via defensive character-by-character validation (lines 35-40), which enforces a purely ASCII alphanumeric set with minor symbols. There are no visually identical characters with differing byte patterns because there are no multi-byte Unicode characters allowed.

Second, the codebase contains a silent data-loss hazard: client-side file-key derivation is hard-coded to use the username as a salt. In client-side encryption architectures, usernames cannot merely be treated as mutable server keys; they are the deterministic inputs for client keys. In Arkfile, the client derives the Account-KEK via Argon2id using the deterministic salt: `SHA-256("arkfile-account-key-salt:{username}")`. If the server permits any discrepancy in lookup case sensitivity during logins without enforcing strict canonical lowercase-only canonicalization on the client before deriving keys, users can log into their accounts but permanently lose access to their file contents. For example, if a user registers as `alice` and later logs in on a mobile device as `Alice`, and the server uses case-insensitive lookup (matching `alice`'s row), the login succeeds. However, the client derives the decryption salt using `Alice`'s byte string. Because the salt changed by exactly one byte (`A` vs `a`), the derived KEK is different; decryption fails with a signature mismatch on download, locking the user out of their data. Therefore, allowing case-insensitive login match without forcing client-side lowercase username canonicalization is an absolute lockout hazard. The ideal best practice is strictly lower-casing usernames at the client edge prior to both registration and login, ensuring the salt is completely stable.

Third, symbol collisions (impersonation vectors) remain an active concern (e.g. registering `john_doe` or `john-doe` when `john.doe` is already registered). To enforce security while retaining a wide range of alphanumeric layouts, we recommend a robust dual-layer strategy:
1. **Tightening Punctuation Adjacency:** In `utils/username_validator.go`, the code blocks identical consecutive symbols (e.g., `..`, `--`, `__`, `,,`) but permits mixtures like `john-.doe` or `john._doe`. We should update the validator to block any consecutive punctuation characters, even if they are different (no character in `[._-,]` can be immediately adjacent to another character in that set). This enforces visual and typographical hygiene and protects username parsing routines.
2. **Case and Symbol Folding uniqueness at registration:** To block registration-time collision impersonators, the server should map each prospective username to a fully folded lowercase, symbol-stripped representation (e.g., `John.Doe` or `john-doe` both fold to the alphanumeric search index input `johndoe`). When a user attempts registration: the server folds the name, checks the table for an existing row carrying that folded index (which maps to `john.doe`), and rejects the collision if present. This allows the genuine user to utilize original punctuation and spacing while guaranteeing that visual homographs and symbol-scrambled duplicates are locked out. Standard lookups and key derivations proceed with the original stored username.
