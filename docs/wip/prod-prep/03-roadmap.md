# Arkfile Post-Launch Feature Roadmap

This document collects the aspirational, product-scope work that was previously mixed into the security and deployment review at `docs/wip/archive/review/remaining-issues.md`. None of these are security or privacy gaps in the current codebase, and none of them should gate a first production launch; they are deliberately separated here so that the live security work (`01-security.md`) and the deployment hardening checklist (`02-prod-hardening.md`) stay focused on what actually blocks going live. Everything below is forward-looking product direction to plan and sequence after a first prod cut is stable. These items were carried forward from the developer-aspiration notes and were not re-verified against implementation status, since they are largely net-new features rather than fixes.

## Stronger and more flexible second-factor authentication

A natural early post-launch enhancement is hardware security key support, letting users register YubiKeys, Nitrokeys, and similar devices as a second factor, including the ability to privately label each device so a user can distinguish their own keys. Closely related is allowing more than one second-factor method per account, initially capped at a small number (around three) to keep the recovery and account-state model manageable: for example a TOTP app such as Ente Auth or Bitwarden Authenticator plus one or more hardware keys. This work touches the existing TOTP flows, the account/credential data model, and the recovery story (backup codes already exist and are hashed), so it should be designed as a coherent multi-factor enrollment and recovery feature rather than bolted on one method at a time. It is meaningful for user trust but is an enhancement to an already-working 2FA system, not a prerequisite for launch.

## Multi-node "cluster mode" for production deployments

A larger ambition is to make the production deployment system encourage redundant, highly-available setups out of the gate: two or more application server backends that talk to each other via rqlite Raft consensus, sitting behind a load balancer. This changes the TLS and DNS assumptions baked into the current single-node design, where Caddy terminates TLS on the same host with a deSEC DNS-01 challenge; a clustered, load-balanced topology may need a different certificate-management approach (shared certs, a fronting load balancer that terminates TLS, or per-node internal certs) and likely changes to the Caddy and deSEC configuration. The deployment scripts (`prod-deploy.sh`, `prod-update.sh`, and the setup scripts) would need to support provisioning and updating a node that is one member of a cluster rather than a standalone instance. If the test and local deploy paths gain a limited version of this multi-node capability as a side effect, that is a bonus but not a requirement. This is a substantial deployment-architecture project and should follow, not precede, the hardening in `02-prod-hardening.md`.

## Container and orchestration strategy

Tied to the cluster-mode ambition is revisiting the earlier plan around Podman containers and Podman Compose, and considering whether Kubernetes would make the "two-or-more-node" topology easier to operate. Per project policy, container work uses Podman (rootless, with explicit developer consent for each invocation) rather than Docker. This is best treated as enabling work for cluster mode: decide the orchestration approach in service of the redundancy goal rather than as an end in itself, and weigh it against the added operational complexity for self-hosters who will run a single node.

## Payments and billing build-out

Before a full public launch, the payments system needs further development, including BTCPayServer integration. The storage-credits and usage-metering core already exists in the codebase (billing scheduler, rates, sweep, and gift mechanics), so this is primarily about adding real payment rails on top of that foundation. The stated target is to launch with support for Bitcoin and Monero alongside conventional credit-card payments. This is product/business-readiness work; it should be planned alongside the launch timeline but kept distinct from the security and deployment-hardening tracks so that neither blocks the other. For this work, we should reference one previous planning document for context: `/docs/wip/storage-credits-v2.md`.

## Sequencing note

The rough intended order is: build out payments so that at the very least bitcoin can be accepted through btcpayserver (ideally also monero and credit card payments through the stripe extension); then layer in stronger multi-factor authentication; and finally take on multi-node cluster mode and the associated container/orchestration decisions, since those are the largest architectural lift and benefit most from a stable single-node baseline to build on.
