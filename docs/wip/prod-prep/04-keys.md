# Server Secrets, Key Rotation, and OPAQUE Re-registration

This document inventories every server-held secret in Arkfile, maps each one to a trust layer, and lays out what rotation infrastructure exists today versus what should be built before a real production deployment with paying customers. It also specifies a full-greenfield rename that removes the opaque "Tier" numbering from the codebase, and it details how a user whose authentication record must be rebuilt can be reconnected to all of their existing files, shares, MFA enrollment, balance, payments, and contact info without re-uploading anything.

The guiding constraint throughout is the one in `docs/AGENTS.md`: OPAQUE authentication and file encryption are cryptographically separate and never interact. That single property is what makes most of this tractable, and it is restated here because almost every rotation decision depends on it.

## Current state: server secret inventory and trust layers

Arkfile holds four distinct layers of server-side secret material. The layering matters because it determines what an attacker gains from each kind of leak, and therefore what a rotation actually needs to accomplish.

| Layer | Where it lives | Members | What a leak enables | Rotation today |
|---|---|---|---|---|
| Envelope master key | `ARKFILE_MASTER_KEY` in `etc/secrets.env` (env var, file mode 0640) | The single 32-byte key that wraps every row in `system_keys` | Decrypt all operational and server-identity secrets | None |
| Operational secrets | `system_keys` table, wrapped by the envelope master key | `jwt_signing_key_temp_v1`, `jwt_signing_key_full_v1`, bootstrap token | Forge or accept sessions; forced re-login resolves it | None (documented as not yet implemented) |
| Server-identity secrets | `system_keys` table, wrapped by the envelope master key | `opaque_server_private_key`, `opaque_oprf_seed`, `entity_id_master_key` | Server impersonation in future logins; EntityID correlation | None |
| User-secret master | `/opt/arkfile/etc/keys/user-secret-master.bin` file (mode 0400), HKDF-expanded | `mfa_user` and `contact_info` subkeys wrapping TOTP secrets, WebAuthn credential blobs, and contact info | Decrypt MFA and contact info | Fully built (prepare/apply with re-encryption) |

Separately there are TLS certificates: the public certificate is issued and renewed by Caddy via Let's Encrypt, and the internal rqlite and SeaweedFS certificates are generated by the setup scripts. The key health monitor in `monitoring/key_health.go` watches all of these for age and permissions but does not rotate any of them.

The user-secret master is the only layer with working rotation today. The tooling performs a mandate-gated, offline, transactional re-encryption of all wrapped rows under a freshly generated master, with a backup of the old master and a post-rotation decrypt verification sample. It is the hardest rotation to get right and it is done well. Everything else in the table currently has no rotation path.

### Files and shares never need to be re-uploaded

The most important reassurance for a production launch is that no server-side key rotation of any kind can ever force users to re-upload files or rebuild shares. Each file's content is encrypted with a random per-file File Encryption Key, and that key is wrapped by a client-derived Account Key or Custom Key produced by Argon2id over the user's password entirely on the client. Share envelopes are wrapped by a client-derived Share Key. None of these wrapping keys are derived from OPAQUE outputs, the JWT keys, the envelope master key, or any server secret. Rotating server secrets is therefore purely a server-side and session-continuity concern; ciphertext at rest is never touched. This holds even for the most disruptive case, OPAQUE server key replacement, which is covered in detail below.

## Greenfield rename: removing the "Tier" numbering

The code currently uses "Tier3" pervasively as the in-situ name for the user-secret master subsystem, and `docs/security.md` describes a three-way "Tier-1, Tier-2, Tier-3" split. This conflicts with the `docs/AGENTS.md` rule against naming functions, variables, and comments after planning-document tier or phase numbers that require an external lookup to understand. Because Arkfile is greenfield with disposable test and beta data, the fix is a clean, total rename rather than any compatibility shim. The three conceptual layers become descriptive names used consistently in code and docs:

- the secrets wrapped by `ARKFILE_MASTER_KEY` are the operational secrets (JWT signing keys, bootstrap token) and the server-identity secrets (OPAQUE private key, OPRF seed, entity ID master),
- the filesystem-isolated master and its subkeys are the user-secret master.

The Go symbol rename maps `Tier3` to `UserSecretMaster` or `UserSecret` according to what reads best in each position: `tier3MasterKey` becomes `userSecretMasterKey`, `Tier3MasterKeyPath` becomes `UserSecretMasterPath`, `LoadTier3Master` becomes `LoadUserSecretMaster`, `DeriveTier3Subkey` and `DeriveTier3SubkeyFromMaster` become `DeriveUserSecretSubkey` and `DeriveUserSecretSubkeyFromMaster`, `ReadTier3MasterFile` and `WriteTier3MasterFile` become `ReadUserSecretMasterFile` and `WriteUserSecretMasterFile`, `SecureZeroTier3` becomes `SecureZeroUserSecretMaster`, `SetTier3MasterForTest` becomes `SetUserSecretMasterForTest`, `Tier3RotationStats` becomes `UserSecretRotationStats`, `ReencryptAllTier3WrappedRows` becomes `ReencryptAllUserSecretWrappedRows`, the `Tier3RotationMandate*` family becomes `UserSecretRotationMandate*`, `ApplyTier3MasterRotation` and its options struct become `ApplyUserSecretMasterRotation`, and `AdminPrepareTier3MasterRotation` becomes `AdminPrepareUserSecretMasterRotation`. The `tier3_rotation_mandates` table becomes `user_secret_rotation_mandates`, and the mandate-file defaults that read `tier3-mandate.txt` or `tier3-rotation-mandate.txt` become `user-secret-rotation-mandate.txt`.

The one rename that is not cosmetic is the HKDF domain-separation label `ARKFILE_TIER3:` in `crypto/user_secret_master.go`, which is mixed into every derived subkey. Renaming it to `ARKFILE_USER_SECRET:` changes the derived keys, so it only takes effect on a fresh deployment where data is regenerated. Under the greenfield assumption this is exactly the intended behavior: the rename lands together with a `dev-reset.sh` (or a clean redeploy of the test and beta environments), and there is no attempt to read data written under the old label. This must be called out in the change so that whoever applies it does not try to hot-patch a populated database and silently lose the ability to decrypt existing MFA and contact rows.

Every `Tier3`/`Tier-3`/`TIER3` identifier, comment, log/error string, schema object, mandate-file default, and operational-doc reference tied to the user-secret master has been replaced with descriptive names, the HKDF label is now `ARKFILE_USER_SECRET:`, and the two rotation source files plus their three test files were renamed on disk. ## Build-out: JWT signing key rotation

**Status: COMPLETE.** Versioned, online, zero-downtime rotation with a verification overlap is implemented for both signing tiers. Each tier is a `jwtKeyRing` holding the active signing keypair plus every version still present in `system_keys` as accepted verification keys (active key first for a fast match). The active version per tier is recorded in a `system_keys` metadata row (`jwt_signing_active_version_temp` / `jwt_signing_active_version_full`), and key rows are versioned (`jwt_signing_key_{temp,full}_vN`). Rotation generates the next version for both tiers, repoints the metadata rows, and reloads the in-memory rings so it takes effect without a restart; previous versions keep verifying until their tokens expire, after which they can be retired. All verification paths (the three audience middlewares, the dual-token MFA-reset middleware, logout/revocation, and export-token validation) try the full verification set. Operated via `arkfile-admin rotate-jwt-keys rotate|retire`, backed by `POST /api/admin/system/rotate-jwt-keys` and `POST /api/admin/system/retire-jwt-key-version`. Unit tests cover the overlap (old token still verifies, new token verifies), retirement (old token then fails, new token still verifies), and the refusal to retire the active version. Builds and `auth`/`handlers` tests pass.

The original design rationale: introduce a `_vN+1` key, begin signing newly issued tokens with it, and keep the previous key in the accepted-verification set until any token signed under it would have expired. Because access tokens are short-lived, the overlap window is small, and the worst case without an overlap is simply that everyone re-authenticates.

## Build-out: envelope master key rotation

**Status: COMPLETE.** Mandate-gated, offline, transactional envelope master rotation is implemented and modeled on the user-secret master prepare/apply pattern. With the service stopped, the apply step decrypts every `system_keys` row under the old master and re-encrypts it under a freshly generated master in a single transaction, then rewrites only the `ARKFILE_MASTER_KEY` line in `secrets.env` (preserving every other line and the file's mode/ownership). Before committing it writes the new master to a root-only `0400` recovery file under `backups/envelope-rotation/` so a failed `secrets.env` swap is always recoverable, and it backs up the whole `secrets.env`. After the swap it performs a thorough full-table verification, decrypting every row under the new master. EntityID master rotation is folded in automatically (see below). Operated via `arkfile-admin rotate-envelope-master prepare|apply`, backed by `POST /api/admin/system/prepare-envelope-master-rotation`, with the `scripts/maintenance/rotate-envelope-master.sh` runbook wrapper.

## Build-out: entity ID master rotation

**Status: COMPLETE.** EntityID master rotation is performed automatically as part of envelope master rotation rather than as a separate command, since regenerating it loses nothing of value except the continuity of the rate-limiting and correlation windows (arguably a privacy improvement). Inside the single re-wrap transaction, every `system_keys` row is re-wrapped under the new master except the EntityID master, which is replaced with fresh random material; on the next service start the new value is read automatically. Because this is a hard reset under a stable key id (we never want old EntityIDs to keep matching), no version-aware loader was needed: instead the meaningless `_v1` suffix was dropped greenfield. The key id `entity_id_master_key_v1` became `entity_id_master_key` and the HKDF derivation label `entity_id_v1` became `entity_id` (both in `logging/entity_id.go`, now sourced from the shared `crypto.EntityIDMasterKeyID` / `crypto.EntityIDKeyType` constants). The stale filesystem path in `monitoring/key_health.go`'s EntityID descriptor was corrected to the `system_keys` key id.

## OPAQUE server keys: stable posture with admin-initiated re-registration

> **STATUS: COMPLETE (as-built).** Admin-initiated OPAQUE re-registration is implemented across the server, both Go CLIs, and the web app. The framing is a routine, periodic OPAQUE credential rotation (e.g. every 1–2 years) rather than an emergency action; nothing about the mechanism assumes a compromise. As-built summary:
>
> - **Schema/model.** `users.requires_reregistration BOOLEAN NOT NULL DEFAULT false` (`database/unified_schema.sql`) with `models.UserRequiresReregistration`, `models.SetUserRequiresReregistration`, and `models.FlagAllUsersForReregistration`.
> - **Handoff token.** `auth.AudienceReregistration = "arkfile-reregistration"`, `auth.GenerateReregistrationToken` (15-minute temp-tier token), and `auth.ReregistrationJWTMiddleware`.
> - **Login branch.** `OpaqueAuthResponse` detects a flagged account at the (empty) record lookup and returns HTTP 409 `account_requires_reregistration` via `respondAccountRequiresReregistration` (`handlers/reregistration.go`), carrying the handoff token, `file_count`, and—when the user owns files—one account-key-encrypted `verifier` sample. It also sets the temp cookie (and clears stale full/refresh cookies) so the browser ceremony carries the token automatically.
> - **Ceremony.** `POST /api/opaque/reregister/response` and `POST /api/opaque/reregister/finalize` (gated by `ReregistrationJWTMiddleware`, rate-limited, CSRF-exempt) reuse the OPAQUE registration internals, replace only the `opaque_user_data` row in place, clear the flag, and continue into the existing MFA flow.
> - **Admin.** `POST /api/admin/users/:username/flag-reregistration` and `POST /api/admin/users/flag-reregistration-all` (`handlers/admin_reregistration.go`) delete only the OPAQUE record, set the flag, and force-logout. CLI: `arkfile-admin flag-user-reregistration --username USER --confirm` or `--all --confirm`.
> - **Clients.** `arkfile-client` (`performReregistration` in `cmd/arkfile-client/reregistration.go`) and the web app (`LoginManager.handleReregistration` in `client/static/js/src/auth/login.ts`) recognize the 409, run the ceremony inline within the same login attempt (no extra login round-trip), and—when the user owns files—confirm the password by test-decrypting the verifier sample before finalizing, aborting with a clear message on mismatch so no wrong-password record is ever bound.
> - **Tests.** Go: `auth/jwt_test.go` (handoff-token claims + `ReregistrationJWTMiddleware` audience acceptance/rejection), `models/reregistration_test.go` (flag helpers, soft-delete handling, `--all` scope), `handlers/reregistration_test.go` (admin single/`--all` flagging deletes only the OPAQUE record and preserves the user row, the 409 verifier-present/absent contract, and the ceremony's unflagged-account rejection), and `cmd/arkfile-client/reregistration_test.go` (pre-finalize password-match verifier accept/reject and fail-safe on missing/incomplete verifier). E2E: `run_opaque_reregistration` in `scripts/testing/e2e-test.sh` reuses the existing user and file to flag, reject a wrong password, re-register inline, and prove the existing file still decrypts. OPAQUE server keys are the one layer that cannot be rotated in place. Each user's stored `opaque_user_data` record is cryptographically bound to the server private key and OPRF seed that existed at registration time. Replacing `opaque_server_private_key` or `opaque_oprf_seed` makes every existing record fail verification, and there is no server-side way to re-wrap those records, because doing so would require the user's password, which the server never possesses. The code reflects this: `auth.SetupServerKeys` generates these once via `GetOrGenerateKey`, and the health monitor annotates the OPAQUE key as never rotated.

The recommended posture for the first production launch is therefore to treat OPAQUE server keys as long-term stable and to provide an admin break-glass procedure for the rare case where they must be replaced (a confirmed or strongly suspected key compromise). This is defensible because a leaked OPAQUE server key does not reveal any past password (OPAQUE resists that) and does not touch file data, so the realistic trigger for replacement is uncommon. A fully transparent rotation is possible later via lazy re-registration, sketched at the end of this section, but it is intentionally out of scope for launch. Re-registration in this document is admin-initiated only; there is no self-service re-registration path.

### Why re-registration reconnects the user to everything

The schema is what makes break-glass clean. Every user-data table is keyed by `username` and foreign-keyed to `users(username)`: `file_metadata`, `user_mfa_credentials`, `user_mfa_backup_codes`, `user_contact_info`, `user_credits`, `credit_transactions`, `payment_invoices`, `file_share_keys`, and the rest. Identity is the username, and it never changes during re-registration. The account file KEK is a deterministic function of the username and the password (`SHA-256("arkfile-account-key-salt:{username}")` fed through Argon2id), so a user who re-registers under the same username and the same password regenerates a byte-identical Account Key, and all account-wrapped files and all account-encrypted metadata continue to decrypt. Custom-password files and share envelopes are independent of the account and are likewise untouched. MFA secrets and contact info are wrapped by the user-secret master (a different layer entirely) and survive as long as their rows survive.

The crucial implementation rule follows from this: break-glass must never delete the `users` row. Deleting it would cascade away files, MFA, shares, credits, and contact info, which is the opposite of the goal, and the deletion would in any case be blocked while `payment_invoices` rows exist because that table uses `ON DELETE RESTRICT`. Instead, break-glass preserves the `users` row and all child rows and replaces only the `opaque_user_data` record. The balance, payment history, MFA enrollment, shares, and contact info are all retained automatically because none of their rows are removed.

There is one user-facing hazard to guard against: if the user re-registers with a different password, their Account Key changes and their account-wrapped files and metadata become undecryptable. Because the server is zero-knowledge it cannot check the password server-side, so the client must verify the password locally before finalizing re-registration by test-decrypting an existing piece of account-encrypted metadata with the freshly derived Account Key. The only material the Account Key protects is file content and file metadata, and metadata is always account-key-encrypted even for custom-password files, so any file the user owns is a usable verifier. Contact info is not a verifier and its presence or absence is irrelevant here, because it is encrypted server-side under the user-secret master (`crypto.DeriveTier3Subkey([]byte("contact_info"))` in `models/contact_info.go`), not under the password-derived Account Key. If the test-decrypt fails, the client must stop and tell the user plainly that the password does not match the one their files were encrypted with, rather than completing a re-registration that would orphan their data. If the user genuinely no longer knows the original password, that is the non-custodial reality already documented in `docs/security.md` (lost password means lost files), and the flow must say so explicitly instead of failing opaquely.

This verification is conditional, because a user who registered but never uploaded a file has no account-key-encrypted data anywhere and therefore nothing to verify against. That case is safe rather than dangerous: with no files there is nothing bound to the old Account Key, so re-registering under any password orphans nothing. The client must therefore make the password-match check mandatory when the user owns at least one file and skip it cleanly when they own none. The decision must be driven by an authoritative server-reported file count (file ownership rows in `file_metadata` are server-side plaintext, not encrypted under the password), never by an empty or failed file-listing response, so that a transient listing error cannot trick the client into skipping verification for a user who actually has files. A future option, if verification is ever wanted for the zero-file case too, is to store an explicit account-key verifier at registration time, but that introduces a new stored per-user value and slightly broadens the offline password-guessing surface, so it is deliberately left out of this design. (One alternative may be to generate a small file client side immediately at registration time for all new users that can serve as a target for account-key verification; It could be encrypted/uploaded just like any other file would be.)

### The core problem to fix: today's failure is indistinguishable from a wrong password

Today, when `opaque_user_data` is missing, the login response path deliberately derives a fake user record (`auth.DeriveFakeUserRecord`) to resist account enumeration, so the client receives a generic authentication failure. In the web app this surfaces as `showError("Authentication failed: " + errorText)` in `client/static/js/src/auth/login.ts`, and in `arkfile-client` it surfaces as the flat `"incorrect password or account not found"` in `handleLoginCommand`. A naive break-glass that simply removed the OPAQUE record would drop the user straight into that dead end. The design therefore introduces an explicit, advertised account state so that a flagged account produces a clear, structured signal that each client can branch on, instead of being routed through the enumeration-resistant fake path.

### Server-side changes

The server needs a persistent account state indicating that re-registration is required, set by the admin break-glass action and cleared when re-registration completes. A boolean such as `requires_reregistration` on the `users` table (defaulting to false) is sufficient, mirroring how `requires_mfa_setup` already gates the post-login flow. When a login reaches the response step for a username whose account carries this flag, the server must skip the fake-record path and instead return a structured response carrying a stable machine-readable code and a short-lived signed re-registration handoff token, modeled on the existing temporary MFA token and the `arkfile-mfa-reset` JWT. That handoff token authorizes only the re-registration ceremony and nothing else, and it expires quickly.

The ceremony itself is a variant of `OpaqueRegisterFinalize` that is gated by the handoff token and by the account flag. Unlike normal registration it does not call `models.CreateUser`, it does not reject the existing username, and it does not create any child rows; it runs the OPAQUE registration steps under the current server keys and replaces the single `opaque_user_data` record in place, then clears `requires_reregistration`. Because MFA enrollment is preserved, the user continues into their existing second factor exactly as in a normal login after the ceremony completes. All of this should reuse the existing multi-step OPAQUE registration endpoints' internals rather than duplicating the protocol code.

The error contract is the part that prevents cryptic dead ends. A small set of stable codes, returned in the structured error body that the clients already parse, lets every client branch deliberately:

- `account_requires_reregistration` (HTTP 409): the account has been flagged by an operator; the response also carries the re-registration handoff token and a human-readable message.
- `reregistration_token_invalid` (HTTP 401): the handoff token is missing, expired, or malformed; the client should restart from a password attempt.
- `reregistration_password_mismatch` (client-side determination): surfaced by the client after the local metadata test-decrypt fails, never by the server.

The following table summarizes the behavior change.

| Scenario | Behavior today | Behavior after |
|---|---|---|
| Login to a flagged account (web) | Generic "Authentication failed" toast | Explanatory screen, guided re-registration, continuation to MFA |
| Login to a flagged account (`arkfile-client`) | "incorrect password or account not found" | Clear message naming re-registration and the next command/step |
| Re-registration with the wrong password | Not applicable | Client stops before finalizing and states the password does not match the files |
| Operator triggers break-glass | No mechanism | `arkfile-admin` command that flags the account and invalidates only the OPAQUE record |

### Web app changes (TypeScript)

The login flow in `client/static/js/src/auth/login.ts` gains a branch that recognizes `account_requires_reregistration` from the step-one response and, instead of showing the generic error, presents an explanatory screen along the lines of: "Your account needs a one-time security re-registration. Your files, shares, and settings are safe. Re-enter your existing account password to reconnect." The screen then runs a re-registration ceremony that reuses the OPAQUE registration logic in `client/static/js/src/auth/register.ts` against the new ceremony endpoint, carrying the handoff token. Before finalizing, when the server reports the user owns at least one file, the client derives the Account Key and test-decrypts existing account-encrypted metadata to confirm the password matches; on mismatch it shows the `reregistration_password_mismatch` message and does not finalize. When the user owns no files the check is skipped because nothing is bound to the Account Key. On success the user proceeds into the already-enrolled MFA flow and on to a full session. New typed error variants and user-facing strings replace the raw passthrough of server text so that no screen ever shows a bare status code.

### arkfile-client changes (Go)

The login command in `cmd/arkfile-client/main.go` (`handleLoginCommand`) must recognize the `account_requires_reregistration` code rather than collapsing it into `"incorrect password or account not found"`. This fits the existing error-classification pattern in `makeRequestWithSession`, which already maps specific server conditions to typed errors such as `errAccountDisabled` and `errQuotaExceeded`. On detecting the flagged state the client explains what is happening and runs the re-registration ceremony with the same password it already holds in memory, then performs the same conditional local password-match verification (by deriving the Account Key and test-decrypting known account metadata before finalizing) whenever the server reports the user owns at least one file, skipping it when they own none. The messages must name the situation and the next step clearly, for example stating that an operator has flagged the account for re-registration, that files and settings are intact, and that the same password must be used to retain file access.

Key files to review and update:

- `cmd/arkfile-client/main.go` (`handleLoginCommand` and the response/finalize handling)
- the request error-classification helper in the same package (extend the typed-error mapping to the new code)
- the client OPAQUE registration helpers used at signup (reused for the ceremony)
- the account-key and metadata helpers used for the local password-match check
- the CLI manual or help text describing the re-registration prompt

### arkfile-admin changes (Go)

The operator-facing command is modeled directly on the existing `reset-user-mfa` command in `cmd/arkfile-admin/mfa_reset_commands.go`, which already demonstrates the right ergonomics: it loads and displays on-file contact info first, requires out-of-band identity verification, requires an explicit `--confirm`, requires `--acknowledge-no-contact-info` when no contact info exists, force-logs-out the user, and prints an instructive closing message. The new command (for example `arkfile-admin flag-user-reregistration --username USER --confirm`) sets `requires_reregistration` on the account and invalidates the `opaque_user_data` record without touching any other row, force-logs-out the user's sessions, and prints guidance to relay to the user: that they should log in normally with their existing password, that their files and settings are safe, and that they must use the same password they originally used. Because this is the break-glass action accompanying an OPAQUE key replacement, the command should be runnable in a mode that flags every account at once for the global-compromise case, in addition to the single-user form.

Key files to review and update:

- a new command file in `cmd/arkfile-admin/` parallel to `mfa_reset_commands.go`
- `cmd/arkfile-admin/main.go` (command registration and dispatch)
- a new admin handler in `handlers/` parallel to `admin_rotation.go` and the MFA reset handler, plus its route in `handlers/route_config.go`
- the server-side state mutation that sets the flag and clears the OPAQUE record without cascading
- `docs/security.md` (incident procedure for OPAQUE key compromise, replacing the current delete-and-force-re-registration sketch with this guided flow)

### Lazy re-registration (future, out of scope for launch)

A fully transparent rotation that avoids any operator action is possible later by supporting two server key versions at once. Each `opaque_user_data` row records the key version it was registered under; the server keeps both the old and new key material loaded and verifies each login against the row's version; and on the next successful login under the old version the client, which transiently holds the password, runs a registration ceremony under the new version and submits the new record, after which the row is marked migrated. Once all rows are migrated or a deadline passes, the old key is retired. This is more involved than break-glass because it requires a key-version column, dual-key loading in `auth.SetupServerKeys`, a verification path that selects key material per user, and a client-driven post-login migration step. It is recorded here as the eventual direction but is deliberately deferred; the launch posture is stable keys plus the admin break-glass flow above.

## Suggested sequencing

JWT signing key rotation should be built first because it is the rotation most likely to be exercised routinely and carries the least risk. The envelope master key rotation should follow, since it protects the highest-leverage secret and its tooling can be modeled directly on the existing user-secret master flow. EntityID master rotation is a small addition once the envelope re-wrap plumbing exists. The greenfield rename should land on its own, paired with a fresh deploy so the HKDF label change takes effect cleanly. The OPAQUE break-glass re-registration flow is the largest piece because it spans the server, the web app, and both Go CLIs, and it should be scheduled deliberately with the error-contract and the client password-match verification treated as first-class requirements rather than afterthoughts, so that a flagged account is always met with clear guidance instead of a bare failure.
