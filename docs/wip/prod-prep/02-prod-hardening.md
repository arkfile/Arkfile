# Arkfile Production Deployment Hardening Checklist

This document is the deployment- and operational-hardening work needed to take Arkfile from a working test/demo deployment to something trustworthy for a real-world production launch. It was derived from a code-verified review on 2026-06-08 of `scripts/prod-deploy.sh`, `scripts/prod-update.sh`, `scripts/setup/build.sh`, `scripts/setup/05-setup-seaweedfs.sh`, and the related config/templates, plus the earlier reviews in `docs/wip/archive/review/remaining-issues.md`. Each item below states its verification status. Items marked "verified" were read directly in the live source on this date with the cited file/line; items marked "from the June 6 review, re-verify before acting" come from the earlier review of files this pass did not re-read (notably the rqlite build script), and should be re-confirmed before code changes. The crypto core is in good shape (OPAQUE hardening, AEAD path-binding, and most sharing residuals are done), so the dominant pre-launch risk now lives in this deployment and operational tooling rather than in the application's cryptography.

Note on one overlap with `01-security.md`: the highest-priority deployment item, binding Arkfile to loopback, is also the one genuinely exploitable in-codebase gap, so it appears in both documents. It is restated here only because the deploy script half of that fix (writing `HOST=127.0.0.1`) lives in this tooling.

## First batch: must-fix before a real production launch

Bind Arkfile to loopback in production (verified). As detailed in `01-security.md`, `main.go` binds `e.Start(":" + port)` and `e.StartTLS(":"+tlsPort, ...)` on all interfaces and ignores the already-existing `config.Server.Host` field. The deploy-tooling half of the fix is to make `main.go` honor `cfg.Server.Host` and to have `prod-deploy.sh` write `HOST=127.0.0.1` into `secrets.env` alongside the existing `PORT`/`TLS_PORT` lines, so the production app is reachable only via Caddy over loopback.

Fail closed when firewall posture is unknown (verified). In `prod-deploy.sh` the `configure_firewall` function (around line 194) configures firewalld on RHEL or ufw on Debian when present, but if neither tool is detected it only prints `WARNING: No supported firewall tool detected; continuing without firewall changes` and proceeds. Combined with the app's all-interfaces bind, this silently produces an internet-exposed instance on hosts without a recognized firewall. The script should halt in that case unless the operator explicitly passes an override flag (for example `--external-firewall-confirmed`). A post-start listening-socket check (confirming Arkfile, rqlite, and SeaweedFS are not externally reachable) would be a valuable additional guard.

Set a restrictive umask before writing any secrets (verified). `prod-deploy.sh` writes `secrets.env`, `rqlite-auth.json`, `seaweedfs-s3.json`, and `/var/lib/caddy/caddy-env` with shell redirection and only afterward applies `chmod 640`/`600` (for example the `secrets.env` heredoc near line 276 is followed by `chmod 640` near line 419). With a typical `umask 022` there is a brief window where these files exist at mode 0644 before the chmod lands. The script should set `umask 077` near the top before any secret material is written, then relax specific files to `0640` only after ownership is set.

Remove argv-based secret handling on the remaining paths (verified, partially already addressed). The deSEC token path is largely fine already: `prod-deploy.sh` prompts interactively when `--desec-token` is omitted and warns loudly when it is passed via the CLI (around lines 884-891); the remaining work there is to deprecate or remove the documented `--desec-token <token>` flag in favor of interactive entry or a `--desec-token-file`, rather than to fix an always-on exposure. Two argv exposures do remain: the rqlite readiness check runs `curl -u "arkfile-db:${RQLITE_PASSWORD}" http://localhost:4001/status` (around line 643), placing the rqlite password in the process command line for the life of that curl; and the printed bootstrap instructions tell the operator to run `bootstrap --token $(sudo cat /opt/arkfile/etc/keys/bootstrap-token.bin)` (around line 1192), which puts the bootstrap token in shell history and argv if copied verbatim. The fix is stdin/file-based secret handling, including adding an `arkfile-admin bootstrap --token-stdin` path so the printed instructions can avoid argv entirely, and using a netrc or stdin mechanism for the rqlite readiness probe.

Make `prod-update.sh` fail and roll back safely (verified). `prod-update.sh` backs up the current binaries before overwriting them (around lines 313-326), but if the new Arkfile or Caddy readiness checks fail it simply `exit 1`s (around lines 431 and 448) after having already stopped services and installed new binaries, static assets, systemd units, and a regenerated Caddyfile, leaving a stopped or half-updated instance with no automatic restore. It also states that data/config/keys are preserved, which is true of files, but starting a new binary can still run application-level schema migrations (see `runSchemaMigrations` in `main.go`), so an update can mutate database state indirectly. A production update flow should take a pre-update rqlite snapshot/export (or require an explicit backup confirmation), back up static assets and service/proxy files alongside the binaries, and install a rollback trap that restores the prior version and restarts the old binary if readiness checks fail.

## Second batch: supply-chain and build-integrity hardening

Pin the Caddy build (verified). `prod-deploy.sh` installs xcaddy with `@latest` (`go install github.com/caddyserver/xcaddy/cmd/xcaddy@latest`, around line 496) and builds Caddy with `--with github.com/caddy-dns/desec` and no version pin (around line 515). Two deployments at different times can therefore build different proxy code. Pin xcaddy, the Caddy base version, and the deSEC DNS plugin version, and record those versions in a deployment manifest for reproducibility.

Replace SeaweedFS MD5 with a pinned SHA-256 (verified). `scripts/setup/05-setup-seaweedfs.sh` downloads the release tarball and then downloads the matching `.md5` from the same GitHub release URL and compares (around lines 57-72). Because the checksum is fetched from the same place as the artifact, this is a corruption check rather than a supply-chain integrity check: an attacker able to substitute the tarball can substitute the checksum too. Replace it with a repo-pinned SHA-256 digest stored alongside the other dependency hash material (for example `config/dependency-hashes.json`), and fail closed if the downloaded artifact does not match. The SeaweedFS cache directory is already `/var/cache/arkfile/seaweedfs` (line 16), so the separate "build cache in the runtime tree" concern does not apply to SeaweedFS.

Add a production build mode that refuses dependency mutation (verified). `scripts/setup/build.sh` runs `go mod tidy` and `go mod vendor` whenever `go mod download` or the vendor-consistency check fails (around lines 113-176). That is convenient during development but non-standard for a production deploy/update, because it can alter `go.mod`, `go.sum`, or `vendor/` on the VPS and mask unreviewed dependency drift. A production build mode should use the checked-in vendor/lock state and fail closed if module or vendor state is inconsistent rather than silently mutating it. The Bun side is already correct here: the frontend build uses `bun install --frozen-lockfile` (around line 353), which refuses to install on a package.json/bun.lock mismatch, and that pattern is the model to mirror on the Go side.

Add vulnerability scanning and an SBOM to the build (verified absent). `scripts/setup/build.sh` runs neither `govulncheck` nor `bun audit` and emits no SBOM. Adding these (failing the build on high-severity findings, emitting an SBOM artifact) closes a supply-chain visibility gap and is a reasonable launch prerequisite for software that will hold other people's encrypted data.

## Second batch: update-path and least-privilege correctness

Use `rsync --delete` (or an explicit clear) for static assets in `prod-update.sh` (verified). The update overlays new static files with `cp -r "$BUILD_CLIENT/static/." "$ARKFILE_DIR/client/static/"` (around line 337), which never removes files that no longer exist in the new build. Stale JavaScript, source maps, or obsolete assets can remain reachable after an update. Use `rsync --delete` where available, or explicitly clear the managed static directories before copying.

Make systemd service-file installation and Caddy validation fail closed in `prod-update.sh` (verified). The update copies service files with `cp ... /etc/systemd/system/ 2>/dev/null || true` (around lines 343-346), so a failed copy is swallowed and an update can appear successful while leaving stale service hardening or execution settings in place; expected service files should install with hard-failure semantics. Separately, the update treats Caddy validation as a warning and skips it entirely when it cannot read the deSEC token (around lines 405-410), whereas `prod-deploy.sh` already fails closed on Caddy validation (around line 593). The update path should also fail closed before restart if the generated Caddyfile does not validate, and treat missing token state as an error unless it can prove the current Caddyfile does not need the token.

Remove the Caddy backend `tls_insecure_skip_verify` (from the June 6 review, re-verify before acting). The June 6 review noted that Caddy proxies to Arkfile on localhost with `tls_insecure_skip_verify`. Because it is loopback it is not the highest risk, but it is non-standard; once Arkfile is loopback-bound, either proxy plain HTTP internally or configure Caddy to trust Arkfile's generated internal CA/certificate instead of skipping verification. Re-read `Caddyfile.prod` before changing.

Revisit the broad `/opt/arkfile` ownership model (verified in part). `prod-deploy.sh` repeatedly `chown -R`s the full tree to `arkfile:arkfile` and `verify_ownership` even treats any root-owned file as a deployment failure (around lines 114-127). Runtime systemd hardening mitigates this, but a more standard least-privilege model would keep binaries, service templates, static assets, and immutable schema/config material root-owned and grant the `arkfile` user ownership only over runtime-write paths and the specific secret/key paths it must read or write. NOTE: Anything we change along these lines must NOT break dev-reset.sh which is critical to local dev testing.

Address the rqlite build-path rough edges (from the June 6 review, re-verify before acting). The June 6 review found that `scripts/setup/06-setup-rqlite-build.sh` pins a version and expected commit (good) but temporarily chowns parts of the runtime tree to `$SUDO_USER` for build-cache access and only partially restores ownership, and suggested preferring a dedicated build cache outside the runtime directory and clarifying `GOTOOLCHAIN` handling (likely `GOTOOLCHAIN=local`). This pass did not re-read that script, so confirm these specifics before acting.

## Lower-stakes cleanups

A few small items round this out (verified where cited). `prod-deploy.sh` still tells operators encountering an existing deployment to "use a future prod-update.sh" (around line 983) even though `prod-update.sh` now exists; update that message. Production deploy/update output should avoid printing commands that place secrets directly in shell command lines (this ties into the bootstrap-token and rqlite items above). The June 6 review also noted the Caddy systemd unit grants write access to `/etc/caddy` even though Caddy should only need to read its config; re-verify the unit before changing it.

## Recommended sequence

Land the first batch before any real launch: loopback bind (with the `HOST=127.0.0.1` write), fail-closed firewall, `umask 077` before secret writes, removal of the remaining argv secret exposures, and a safe fail/rollback path in `prod-update.sh`. Then do the supply-chain and build-integrity batch (pin Caddy/xcaddy/deSEC, SHA-256 for SeaweedFS, a no-mutation production build mode, govulncheck/bun audit/SBOM). Finally clean up the update-path correctness items, the Caddy backend verification, the ownership model, and the small cosmetic fixes.
